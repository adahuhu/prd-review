# PRD 内审检查清单（v2.5.1）

> 配套《PRD 文档标准规范 v2.5》使用
> 审查时按 严重 → 重要 → 建议 → 逻辑校验 顺序逐项检查
> 使用说明：✅ Pass / ⚠️ Partial / ❌ Fail / N/A 不涉及
>
> **嵌入内容预扫描机制（v2.5.0 新增）**：审核正式开始前，Skill 会自动扫描文档中的非纯文本内容（嵌入式 Smart Sheet / Excel、图片、流程图等）。无法通过 API 自动读取的内容会在预扫描报告中列出，PM 可选择「补充内容」「跳过（标记待确认）」或「只审正文」。**审核前请注意查看预扫描报告**，避免因内容缺失导致审核不完整。

## 评分规则

### 扣分权重

起始分 100 分，扣到 0 为止。N/A 项不计分。

| 级别 | ❌ Fail 扣分 | ⚠️ Partial 扣分 |
|------|-------------|-----------------|
| 严重 | -8 分 | -4 分 |
| 重要 | -4 分 | -2 分 |
| 建议 | -1 分 | -0.5 分 |

总分四舍五入取整。

### 通过标准

| 结论 | 条件 |
|------|------|
| ✅ 通过 | ≥ 80 分 且 零质量红线违反 |
| ⚠️ 有条件通过 | 60-79 分 且 零质量红线违反（需修复严重项后可推进） |
| ❌ 不通过 | < 60 分 或 有质量红线违反 |

## A. 严重级（必须修复，否则阻塞开发）

### A1. 文档完整性

| 编号 | 检查项 | B 型 | A 型 | C 型 | 结果 | 扣分 |
|------|--------|------|------|------|------|------|
| A1-1 | 不缺少背景/现状说明 | 必检 | 必检 | 必检 | | |
| A1-2 | 功能描述清晰无歧义 | 必检 | 必检 | 必检 | | |
| A1-3 | 涉及数值的地方有具体值 | 必检 | 必检 | 必检 | | |
| A1-4 | 无模糊词（需上下文判定） | 必检 | 必检 | 必检 | | |

> **A1-4 模糊词上下文判定规则：**
> 发现模糊词（适当、合理、若干、等、类似、尽量、必要时、优化、完善）后，检查**同一句或相邻两句（下文 2 句内）**是否有量化补充：
> - 有具体数值（如"默认 1000 QPS""前 50 条"）→ ✅ 已解释，不扣分
> - 有限定范围（如"不超过 XX""在 XX 范围内"）→ ✅ 已解释，不扣分
> - 有操作条件（如"用户手动配置后""达到阈值时触发"）→ ✅ 已解释，不扣分
> - 无任何量化补充 → ❌ 裸模糊词，扣分
>
> **示例：**
> - ❌ "系统会进行**合理**的限流" — 无补充 → 扣分
> - ✅ "系统会进行**合理**的限流，默认上限 1000 QPS" — 有具体值 → 不扣分
> - ✅ "**必要时**触发告警，当连续 3 次失败或响应超 5 秒时" — 有操作条件 → 不扣分

### A2. 状态与数据（A 型必检，B/C 型按需）

> **A2-1 状态机前置判定（v2.4.1 新增，v2.5.1 合并为一项）**：
> 状态机检查仅在文档**实际涉及业务对象的状态流转**时适用。以下情况标记 N/A（不检查 A2-1）：
> - 文档中无业务对象（订单/计划/单据等）在生命周期中经历状态变化
> - "状态"仅指 UI 展示状态（空状态、加载状态、无数据状态）而非业务状态
> - 功能为纯数据计算/展示/汇总，无审批、流转、生命周期管理
> - **判定方法**：看文档是否提及某个业务对象从 A 变为 B 的逻辑链路。若有且文档未定义状态机 → 扣分；若仅是"某条件下显示某结果"的计算逻辑 → N/A
>
> **A2-2 触发条件（三个条件同时满足才检查，否则 N/A）**：
> 1. 文档中明确提到「新增字段」「新增列」「新建字段」或类似表述
> 2. 该字段在现有系统中不存在（需对比上下文判断）
> 3. 字段需要有业务含义定义（不是简单的数据搬运/同步/展示）
>
> 以下场景**不触发** A2-2（标记 N/A）：
> - 数据同步/传输：已有字段从 A 系统推送到 B 系统
> - 数据展示优化：页面增加已有字段的可视化展示
> - 数据格式调整：已有字段的显示格式变更
> - 快照/缓存数据：基于已有数据生成的快照

| 编号 | 检查项 | 结果 | 扣分 |
|------|--------|------|------|
| A2-1 | 状态机完整性：涉及状态流转已定义操作驱动型状态机，包含完整字段（当前状态→执行操作→流转后状态→角色限制→说明），终态已标注 | | |
| A2-2 | 新增核心字段有字段定义（仅新增字段时检查，数据传输/展示场景 N/A） | | |

### A3. 影响范围（B 型按需提示，C 型必检，A 型按需）

> **B 型影响范围特殊规则**：B 型优化中影响范围为**建议项**（不扣分），当检测到涉及跨模块/跨系统/存量数据时在报告中提示补充，但不影响评分。这是因为大部分 B 型优化影响范围有限（改个按钮、加个筛选），且规范原文标注「若不涉及可删除」。

| 编号 | 检查项 | B 型 | A 型 | C 型 | 结果 | 扣分 |
|------|--------|------|------|------|------|------|
| A3-1 | 影响范围已评估 | 按需（提示） | 按需 | 必检 | | |
| A3-2 | 存量数据处理方式已说明（保留/回填/置空/需修复） | 按需 | 按需 | 必检 | | |
| A3-3 | C 型已判断是否有后续正式需求 | — | — | 必检 | | |

## D. 逻辑校验（所有类型必检）

> 逻辑校验关注需求本身的业务合理性，而非文档结构完整性。
> 不确定的问题**不给建议**，仅标注"待确认"。

| 编号 | 校验维度 | 校验内容 | 结果 | 说明 |
|------|----------|----------|------|------|
| D1-1 | 前后一致性 | 功能描述中的前置条件与后续业务规则是否自洽（如"提交前需审核通过"与"提交后进入待审核"矛盾） | | |
| D1-2 | 字段依赖 | 同一字段在不同章节的定义/取值范围/约束是否一致（如"金额"某处写"必填"某处写"可为空"） | | |
| D1-3 | 流程闭环 | 操作流程是否有未处理的出口（如只有"审核通过"路径，缺少"驳回""撤回"的处理） | | |
| D1-4 | 状态可达性 | 状态机中每个状态是否都可通过合法操作到达，是否有不可达的孤立状态 | | |
| D1-5 | 数据引用 | 功能描述中引用的字段/枚举值是否在字段定义/数据字典中有对应说明 | | |
| D1-6 | 条件互斥 | 多个条件规则之间是否存在逻辑冲突（如"A和B互斥"同时又有"必须同时满足A和B"） | | |
| D1-7 | 边界值 | 涉及数值/列表/分页等功能时，是否定义了上下界和空值处理 | | |
